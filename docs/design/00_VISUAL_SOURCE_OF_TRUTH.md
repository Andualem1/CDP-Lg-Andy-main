# DEC / WHH CSF+ CSO Learning Hub — Visual Source of Truth

## 1. Document purpose

This file defines the visual continuity anchor for the DEC / WHH CSF+ CSO Learning Hub Phase 1 application.

It is the first design-control document that Codex and any developer must read before implementing or revising user-facing UI.

This document confirms that the platform’s UI/UX direction shall build from the previously presented and stakeholder-liked landing page and course catalogue designs, while improving quality, consistency, accessibility, brand alignment, content accuracy, and implementation readiness.

The purpose is to prevent visual drift and avoid the impression that the product changes direction every time it is presented.

---

## 2. Source reference screens

The current visual source of truth is based on two uploaded reference screenshots:

```txt
Reference 1: Previous landing page / homepage design
Reference 2: Previous course catalogue design
```

These reference screens are not final implementation files. They are continuity references.

They should be used to preserve the visual direction stakeholders already recognized and responded to positively.

---

## 3. Source-of-truth decision

The approved design direction is:

> Continue from the previously presented landing page and learner-facing catalogue visual direction, but refine it into a more consistent, premium, DEC-branded, accessible, and implementation-ready design system.

This means:

1. do not redesign the platform from scratch;
2. do not copy the previous screenshots exactly;
3. preserve the recognizable visual language;
4. improve weak or inconsistent design areas;
5. replace old text/content with current approved product messaging;
6. align colors, components, and layouts with the current Phase 1 product specification;
7. ensure all future screens feel like part of the same product family.

---

## 4. What must be preserved

## 4.1 Overall feeling

Preserve the following feeling from the previous designs:

1. premium;
2. editorial;
3. calm;
4. locally grounded;
5. institutional but modern;
6. learning-focused;
7. friendly to local and grassroots CSOs;
8. visually credible for DEC, WHH, partners, and EU-funded programme stakeholders.

The platform should continue to feel like a serious CSO learning platform, not a generic software dashboard.

---

## 4.2 Landing page structure

Preserve the broad section flow from the previous landing page:

```txt
Top navigation
Hero section with contextual image
Core features
About / platform explanation
Course showcase
Learning experience section
Call-to-action section
Dark institutional footer with partner strip
```

This structure provides continuity with the previous prototype.

Codex may improve the spacing, hierarchy, content, and component styling, but must not replace this with an unrelated generic SaaS landing page.

---

## 4.3 Hero image direction

Preserve the use of a grounded, locally relevant contextual hero image.

The hero image direction should show local CSO learning, group discussion, training, planning, or collaborative work.

The hero should feel Ethiopian/local-context relevant and should avoid generic Western corporate stock imagery.

The previous landing page hero image is considered a strong direction and may be reused if rights/asset availability are confirmed.

If the same image is not technically available, use a visually similar approved asset or placeholder direction that maintains the same local, collaborative, CSO-learning feeling.

---

## 4.4 Editorial headline style

Preserve the strong editorial headline feel from the previous landing page.

The previous use of a large serif-style headline gave the page a distinctive premium identity.

Future landing and major public pages may continue using a strong serif-style or editorial display headline, paired with a clean sans-serif body font.

The typography must be refined for consistency and accessibility.

---

## 4.5 Card-based course browsing

Preserve the card-based approach for course discovery.

The course catalogue should continue to use polished course cards with:

1. course image or visual cover;
2. course title;
3. capacity area/category;
4. short description;
5. duration or lesson count;
6. access/certificate indicator;
7. clear action.

However, the card system must be improved for consistency, spacing, hierarchy, metadata treatment, and mobile responsiveness.

---

## 4.6 Featured course concept

Preserve the featured course concept from the course catalogue.

A featured course section can help highlight priority or recommended courses.

However, the featured course layout must be improved so that it feels aligned with the rest of the catalogue and does not overpower the page or create inconsistency.

---

## 4.7 Dark institutional footer

Preserve the dark footer direction with platform information and partner/donor recognition.

The footer should feel serious, institutional, and polished.

Partner logos should be presented carefully with consistent sizing, sufficient spacing, and responsive behavior.

---

# 5. What must be improved

The previous reference screens are a starting point, not the final quality standard.

The following improvements are mandatory.

---

## 5.1 Brand color alignment

The design must be realigned to DEC’s approved visual direction.

Use the following as the core brand direction:

```txt
Primary Blue: #3B99D4
Accent Green: #91C852
Dark Ink: #111827
Light Background: #F9FAFB
White Surface: #FFFFFF
Border: #E5E7EB
```

Rules:

1. Use DEC blue consistently for primary actions, active states, and institutional accents.
2. Use DEC green sparingly for positive progress, certificates, readiness, and supportive learning indicators.
3. Avoid random blues, greens, yellows, or badge colors.
4. Avoid inconsistent course-card badges.
5. Ensure all color combinations are accessible.

---

## 5.2 Content and messaging replacement

Do not reuse the previous prototype text as final content.

The previous content was useful for demonstration, but current implementation must use updated messaging based on the current Phase 1 product direction.

The platform must now reflect that Phase 1 is:

```txt
An e-learning MVP with a future-ready CSO Learning Hub foundation.
```

Updated messaging should emphasize:

1. practical digital learning for local and grassroots CSOs;
2. structured courses;
3. interactive learning blocks;
4. progress tracking;
5. final tests;
6. certificates;
7. CSO organization and cohort awareness;
8. accessible and mobile-friendly learning;
9. future-ready foundation for wider capacity strengthening.

Avoid outdated, generic, or over-promising text.

---

## 5.3 Course catalogue consistency

The previous catalogue design needs substantial refinement.

Improve:

1. search/filter layout;
2. featured course alignment;
3. course card grid consistency;
4. image proportions;
5. badge styling;
6. metadata hierarchy;
7. access indicator clarity;
8. certificate indicator clarity;
9. spacing between cards;
10. mobile card behavior;
11. typography consistency;
12. empty state design.

The catalogue must feel like a polished course discovery experience, not a collection of inconsistent cards.

---

## 5.4 Course image system

Course cover images must be treated consistently.

Rules:

1. use consistent aspect ratios;
2. avoid overly busy image overlays;
3. avoid unreadable text inside thumbnails;
4. avoid mixed illustration/photo styles without a system;
5. use subtle gradients or overlays only where they improve readability;
6. maintain local relevance;
7. ensure images do not dominate the card text.

If course image content is not yet finalized, use a controlled placeholder image system rather than random visuals.

---

## 5.5 Typography refinement

The design may keep an editorial headline style, but typography must be systematized.

Recommended approach:

1. display/hero headings: editorial serif or strong display font;
2. body/UI text: clean sans-serif;
3. buttons: uppercase or semi-uppercase only if readable and not overused;
4. metadata labels: small but legible;
5. course card titles: strong and readable;
6. dashboard/admin text: clean sans-serif only, unless a display heading is useful.

Avoid inconsistent heading weights and tiny metadata.

---

## 5.6 Navigation refinement

Improve navigation so it feels polished and role-appropriate.

Public navigation should include:

```txt
Home
About
Courses / Catalog
Verify Certificate
Sign In
Register
```

Final labels may be adjusted, but they must be consistent across public pages.

Do not expose:

1. admin links;
2. creator links;
3. route names;
4. technical names;
5. future Phase 2/3 modules as active links.

---

## 5.7 Section rhythm and spacing

Improve vertical rhythm across the landing page.

Each major section should feel intentionally spaced and visually connected.

Avoid:

1. abrupt section transitions;
2. inconsistent section padding;
3. crowded cards;
4. large empty gaps without purpose;
5. weak alignment between text and images.

---

## 5.8 Accessibility and mobile readiness

All refined designs must support:

1. mobile-first responsive behavior;
2. readable text size;
3. accessible contrast;
4. keyboard navigation;
5. meaningful alt text;
6. tappable controls;
7. no horizontal overflow;
8. clear focus states.

The current reference screens are desktop-focused. Mobile behavior must be designed and tested separately.

---

# 6. What must not be copied from the reference screens

Do not copy the following from the old reference designs:

1. old course titles as final content unless they match the seed plan or approved course list;
2. old feature text;
3. inconsistent badge colors;
4. inconsistent course card metadata;
5. overly busy course thumbnails;
6. any outdated terminology;
7. inaccessible small text;
8. visual inconsistencies between catalogue cards;
9. any element that looks like a static image-only prototype rather than an implementable system;
10. any copy that conflicts with the current Phase 1 specification.

---

# 7. Relationship to product specifications

The visual direction in this file must support the approved Phase 1 product specification.

The product specification states that Phase 1 must deliver a working, accessible, mobile-first digital learning platform for local and grassroots CSOs, including public/participant learning, a course catalogue, course player, professional block-based Course Builder, quizzes, certificates, organization/cohort tracking, monitoring, and feedback.

The data model requires a structured learning hierarchy:

```txt
Course → CourseVersion → Module → Lesson → ContentBlock
```

The UI must visibly support this structure in the course player and Build Studio, rather than presenting courses as flat uploaded files.

The learner template specification requires the learner experience to feel like a professional digital course designed for participants, not a folder of uploaded training materials.

The Build Studio specification requires a clean three-column authoring workspace and explicitly protects it from governance, CRM, diagnosis, monitoring, and compliance clutter.

The route map separates the system into public, learner, creator, and admin experience zones. The visual design must also preserve these zones so users never feel they are inside the wrong part of the system.

# 9. Creator, Build Studio, Admin, and Monitoring visual direction

The uploaded reference screens mainly guide the public and learner-facing surfaces.

The creator, Build Studio, admin, and monitoring areas must extend the same brand system but may use a more operational interface style.

---

## 9.1 Creator portal

Creator screens should feel like a calm production workspace.

Use:

1. white cards;
2. structured forms;
3. compact context bars;
4. clear workflow navigation;
5. consistent buttons and status badges.

Avoid:

1. heavy governance;
2. diagnosis screens;
3. large compliance blocks;
4. developer language.

---

## 9.2 Build Studio

Build Studio must be visually premium and highly usable.

It must preserve the required three-column layout:

```txt
Left: Block Library + Course Outline
Center: Course Canvas
Right: Block Configuration
```

It must feel like a professional authoring tool and not like an admin database.

---

## 9.3 Admin portal

Admin screens should be operational and calm.

They do not need to copy the landing page’s editorial style, but they must share the same brand tokens, spacing quality, card design, and polish.

Admin screens must not become CRM-like or donor-management-like.

---

## 9.4 Monitoring dashboard

Monitoring should be clean, decision-oriented, and not overloaded.

It should use KPI cards, filters, concise attention signals, and readable tables.

It must not claim long-term CSO capacity impact in Phase 1.

Before approving any future UI screen, check:

1. Does it still feel connected to the previous stakeholder-liked design?
2. Is it better than the previous design in consistency and polish?
3. Does it use DEC blue and green more coherently?
4. Does it avoid outdated text/content?
5. Does it support the current Phase 1 product scope?
6. Is it mobile-conscious?
7. Is it accessible?
8. Does it avoid developer language?
9. Does it avoid Phase 2/3 or CRM drift?
10. Would we be comfortable showing it to DEC/WHH/EU-funded programme stakeholders?

If the answer to question 1 or 10 is no, the design should not be approved.
